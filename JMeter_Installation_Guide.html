PART 1 — LOAD TESTING FUNDAMENTALS

What is Load Testing?

Load testing is a type of performance testing that simulates multiple users accessing a system at the same time to measure how it behaves under expected and peak conditions.

Think of it like a fire drill for your application — you simulate a real-world scenario before it actually happens, so you can find and fix problems before your users do.

💡 Simple Definition: Load testing answers the question — "Can our system handle the number of users we expect?"

What Load Testing Checks

QuestionWhat It Measures
Can the system handle 500 users at once?Capacity and concurrency
How fast does it respond under load?Response time under stress
Does it crash or return errors?Stability and reliability
Does performance degrade over time?Endurance and memory leaks
How many requests can it handle per second?Throughput and capacity

Why Do We Do Load Testing?

Without Load TestingWith Load Testing
System crashes on launch dayIssues found and fixed before go-live
Users experience slow pagesBottlenecks identified early
No idea of system limitsKnow exact capacity and breaking point
Reactive — fix after failureProactive — fix before failure
Business loss due to downtimeConfidence in system reliability
⚠️ Real World Impact: A system that works fine for 10 users can completely fail for 500 users. Load testing reveals this before your real users experience it.

Types of Load Testing

1. Load Testing

Tests the system under the expected normal load it was designed for.

Example: Your app expects 500 users during business hours — simulate exactly 500 users.

Goal: Verify the system works correctly under normal conditions.

2. Stress Testing

Pushes the system beyond its normal limits to find the breaking point.

Example: Keep adding users (600, 700, 800...) until the system fails.

Goal: Find out at what point the system breaks and how it fails.

3. Spike Testing

Suddenly throws a massive load at the system in a very short time.

Example: Normal load is 200 users — suddenly jump to 2,000 users in seconds.

Goal: See how the system handles sudden unexpected traffic surges.

4. Endurance Testing (Soak Testing)

Runs the system under normal load for a long period (hours or days).

Example: Simulate 500 users continuously for 8 hours.

Goal: Find issues that only appear over time, like memory leaks.

5. Scalability Testing

Tests how well the system scales up as load increases.

Example: Test with 1 server, then 2 servers, then 3 servers at the same load.

Goal: Understand if adding resources improves performance proportionally.

TypeLoad LevelDurationGoal
LoadNormal expected loadShort–MediumVerify normal performance
StressBeyond normal limitsShortFind breaking point
SpikeSudden massive surgeVery shortHandle sudden traffic
EnduranceNormal loadVery long (hours/days)Find time-based issues
ScalabilityIncreasing loadMediumUnderstand growth capacity

Key Performance Metrics

These are the numbers you read and analyze in every JMeter report. Understanding each one is essential for a QA engineer.

1. Response Time

The time from when a user sends a request to when they receive a complete response. Measured in milliseconds (ms).

💡 Think of it like: Ordering food at a restaurant — response time is how long you wait from placing your order to getting your food.
MetricWhat It Means
MinimumFastest response recorded during the test
MaximumSlowest response recorded — watch for extreme outliers
AverageSum of all response times ÷ total requests. Can be misleading if outliers exist.
Median (50th pct)50% of requests are faster than this value. More reliable than average.
90th Percentile90% of requests completed faster than this. 10% were slower.
95th Percentile95% of requests completed faster than this. Commonly used in SLAs.
99th Percentile99% of requests completed faster than this. Represents worst-case users.
💡 Why percentiles matter more than average:
Example: 9 requests take 100ms, 1 request takes 10,000ms.
Average = 1,090ms — looks bad for everyone.
90th percentile = 100ms — most users are actually fine.
The average is misleading; percentiles tell the real story.

2. Throughput (Transactions Per Second — TPS)

The number of requests the system successfully processes per second.

💡 Think of it like: A highway — throughput is how many cars pass through per minute. Higher throughput = more capacity.

Key rule: Always check successful throughput, not just raw throughput.

Successful TPS = Raw TPS × (Success Rate / 100)
Example: 290 TPS × 81.25% = ~236 successful TPS

3. Error Rate

The percentage of requests that failed out of total requests sent.

Error Rate = (Failed Requests ÷ Total Requests) × 100

Common JMeter errors:

Error CodeMeaningLikely Cause
503Service UnavailableServer too busy, thread pool exhausted
504Gateway Time-outBackend took too long to respond
408Request Time-outClient-side timeout waiting for response
ConnectionClosedExceptionConnection closed unexpectedlyServer crashed or closed connection prematurely
HttpHostConnectExceptionCannot connect to hostServer unreachable or connection pool exhausted

4. APDEX Score (Application Performance Index)

A single score from 0 to 1 that measures overall user satisfaction based on response times.

User ExperienceResponse TimeScore Weight
Satisfied≤ 500ms (T threshold)1.0 point
Tolerated500ms – 1500ms (F threshold)0.5 points
Frustrated> 1500ms0 points
APDEX = (Satisfied + (Tolerated / 2)) ÷ Total Requests

5. Concurrent Users (Virtual Users / Threads)

The number of simulated users hitting the system at the same time. In JMeter, each user is called a Thread.

TermMeaning
ThreadsNumber of concurrent virtual users
Ramp-up PeriodTime to gradually reach the target number of users
DurationHow long the test runs at full load
💡 Ramp-up Formula: A good rule of thumb is Ramp-up = Number of Users ÷ 10.
Example: 1000 users → 100 second ramp-up (adds 10 users per second gradually).

6. Network Throughput (KB/s)

The amount of data transferred between the client (JMeter) and the server per second.

MetricMeaning
Received KB/sData downloaded from server per second
Sent KB/sData uploaded to server per second

How Metrics Relate to Each Other

These metrics do not work in isolation — they affect each other. Understanding the relationship is key to diagnosing performance problems.

More Users Added
      ↓
Server Gets Busier
      ↓
Response Time Increases
      ↓
Some Requests Start Timing Out / Failing
      ↓
Error Rate Increases
      ↓
APDEX Score Drops
      ↓
Throughput Plateaus (hits capacity ceiling)
      ↓
System at Maximum Capacity or Failing

What Good Scaling Looks Like

MetricGood Behavior (Scaling Well)Bad Behavior (Bottleneck)
ThroughputIncreases proportionally with usersPlateaus while errors rise
Response TimeStays stable as load increasesJumps significantly with more users
Error RateStays below 1% at all load levelsSpikes as users increase
APDEXStays above 0.85 at all load levelsDrops below 0.70 under load
⚠️ Key Insight: High throughput with a high error rate is NOT good performance. Always check successful throughput alongside error rate. A system processing 500 TPS with 50% errors is only effectively delivering 250 TPS.

PART 2 — JMETER INSTALLATION

Prerequisites

Before installing JMeter, make sure you have the following:

  • Windows machine
  • Internet connection
  • Administrator access
  • At least 4GB RAM (8GB recommended for 500+ users)
  • At least 500MB free disk space
⚠️ Java is required. JMeter is a Java-based application. You must install Java before JMeter will work.
Step 1 — Install Java (JDK)

JMeter requires Java 8 or higher. We recommend Java 11 or Java 17 (LTS versions).

  1. Go to https://adoptium.net (Eclipse Temurin — free and open source)
  2. Select Java 17 (LTS), Windows, x64
  3. Click Download .msi
  4. Run the installer
  5. On the "Custom Setup" screen, make sure these are enabled:
    • ✅ Add to PATH
    • ✅ Set JAVA_HOME variable
  6. Click Next and then Install
  7. Click Finish when done

Verify Java Installation

Open Command Prompt and run:

java -version

Expected output:

openjdk version "17.x.x" ...
💡 If you see "java is not recognized", Java was not added to PATH. Reinstall and make sure to enable "Add to PATH" and "Set JAVA_HOME variable".
Step 2 — Download Apache JMeter
  1. Go to https://jmeter.apache.org/download_jmeter.cgi
  2. Under Binaries, download the .zip file:
    apache-jmeter-5.6.x.zip
  3. Save it to a location you can easily find (e.g., C:\Downloads)
💡 JMeter does not have a traditional installer — it runs directly from the extracted folder. No installation wizard needed.
Step 3 — Install & Configure JMeter

Extract JMeter

  1. Right-click the downloaded apache-jmeter-5.6.x.zip
  2. Select Extract All
  3. Extract to a clean location, recommended: C:\JMeter\
  4. Your folder structure should look like: 
    C:\JMeter\apache-jmeter-5.6.x\
    ├── bin\
    ├── lib\
    ├── docs\
    └── extras\

Set JMETER_HOME Environment Variable (Recommended)

  1. Press Win + S, search for "Environment Variables"
  2. Click "Edit the system environment variables"
  3. Click "Environment Variables" button
  4. Under System Variables, click New
  5. Set:
    • Variable name: JMETER_HOME
    • Variable value: C:\JMeter\apache-jmeter-5.6.x
  6. Click OK
  7. Find the Path variable, click Edit
  8. Click New and add: %JMETER_HOME%\bin
  9. Click OK on all dialogs

Increase JMeter Memory (Important for Large Tests)

For tests with 500+ users, increase the JVM heap size to prevent out-of-memory errors.

  1. Navigate to C:\JMeter\apache-jmeter-5.6.x\bin\
  2. Open jmeter.bat with a text editor (right-click → Edit)
  3. Find the line with HEAP and update it:
set HEAP=-Xms1g -Xmx4g -XX:MaxMetaspaceSize=256m

Save the file. This sets minimum heap to 1GB and maximum to 4GB.

Step 4 — Launch JMeter

Option A — GUI Mode (for building and debugging tests)

  1. Navigate to C:\JMeter\apache-jmeter-5.6.x\bin\
  2. Double-click jmeter.bat
  3. The JMeter GUI window will open

Option B — Command Line (for running actual tests)

jmeter -n -t your_test.jmx -l results.jtl -e -o report_output
⚠️ IMPORTANT: Always run actual load tests in Non-GUI mode (command line). GUI mode consumes significantly more memory and CPU, which affects test accuracy — especially for 500+ users.
ModeWhen to UseCommand
GUI ModeBuilding scripts, debugging, small testsDouble-click jmeter.bat
Non-GUI ModeRunning actual load tests (100+ users)jmeter -n -t script.jmx -l results.jtl
Step 5 — Install JMeter Plugins (Optional but Recommended)

Plugins add extra graphs, samplers, and utilities that make JMeter more powerful.

Install Plugins Manager

  1. Go to https://jmeter-plugins.org/install/Install/
  2. Download jmeter-plugins-manager-x.x.jar
  3. Copy the downloaded .jar file to:
    C:\JMeter\apache-jmeter-5.6.x\lib\ext\
  4. Restart JMeter
  5. Go to Options → Plugins Manager in the JMeter menu

Recommended Plugins to Install

PluginPurpose
3 Basic GraphsResponse Times Over Time, Transactions per Second, Active Threads
Custom Thread GroupsUltimate Thread Group for advanced ramp-up patterns
Throughput Shaping TimerControl exact TPS rate during test
Dummy SamplerUseful for debugging test logic
Step 6 — Create Your First Test Plan

A Test Plan is the top-level container for your entire performance test. Everything lives inside it.

  1. Open JMeter GUI
  2. You will see "Test Plan" already created in the left panel
  3. Click on Test Plan and give it a name (e.g., Search API Load Test)
  4. Right-click Test Plan → Add → Threads (Users) → Thread Group
💡 Test Plan Structure: 
Test Plan
  └── Thread Group (Virtual Users)
        ├── HTTP Request (Sampler)
        ├── HTTP Header Manager (Config)
        ├── Response Assertion (Assertion)
        └── View Results Tree (Listener)
Step 7 — Configure Thread Group

The Thread Group defines how many virtual users run, how fast they ramp up, and how long the test runs.

  1. Click on the Thread Group you just added
  2. Configure the following settings:
SettingDescriptionExample Value
Number of Threads (users)Total concurrent virtual users200
Ramp-up Period (seconds)Time to reach full user count gradually20
Loop CountHow many times each user repeats the testSet to "Infinite" if using Duration
Duration (seconds)How long the test runs at full load600 (10 minutes)
SchedulerEnable to use Duration instead of Loop Count✅ Checked
💡 Ramp-up Tip: Use Ramp-up = Users ÷ 10. For 200 users → 20 second ramp-up. This avoids hitting the server with all users at once and simulates realistic traffic growth.
Step 8 — Add HTTP Request Sampler

The HTTP Request Sampler defines the actual API call or web page request to test.

  1. Right-click Thread Group → Add → Sampler → HTTP Request
  2. Configure the HTTP Request:
FieldDescriptionExample
NameLabel for this request in reportsSearch API
Protocolhttp or httpshttps
Server Name or IPTarget server address192.168.1.100
Port NumberServer port8080
HTTP MethodGET, POST, PUT, DELETEGET
PathAPI endpoint path/api/search
Parameters / BodyRequest parameters or JSON bodyq=test&page=1

Add HTTP Header Manager (if needed)

  1. Right-click Thread Group → Add → Config Element → HTTP Header Manager
  2. Add headers like:
NameValue
Content-Typeapplication/json
AuthorizationBearer <your_token>
Step 9 — Add Listeners (View Results)

Listeners collect and display test results. Add these to your Thread Group:

  1. Right-click Thread Group → Add → Listener
ListenerPurposeWhen to Use
View Results TreeShows each request and response in detailDebugging only — disable during actual load tests
Summary ReportShows aggregate stats per request labelQuick overview during test
Aggregate ReportShows percentiles, min, max, avg per labelDetailed analysis
View Results in TableShows each request as a table rowSmall tests only
⚠️ IMPORTANT: Disable all listeners during actual load tests — they consume memory and affect results. Use the -e -o flags in command line to generate the HTML report after the test instead.
Step 10 — Run the Test

Option A — GUI Mode (small tests / debugging only)

  1. Click the ▶ Start button (green play button) in the toolbar
  2. Watch results in your Listeners
  3. Click the ⏹ Stop button to end the test

Option B — Non-GUI Mode (recommended for actual load tests)

# Basic run
jmeter -n -t your_test.jmx -l results.jtl

# Run and generate HTML report
jmeter -n -t your_test.jmx -l results.jtl -e -o report_output

# Run on remote slaves (distributed testing)
jmeter -n -t your_test.jmx -r -l results.jtl -e -o report_output
FlagMeaning
-nNon-GUI mode
-tPath to your .jmx test script
-lPath to save results (.jtl file)
-eGenerate HTML report after test
-oOutput folder for HTML report (must be empty)
-rRun on all remote slaves (distributed mode)
-RRun on specific slave IPs
Step 11 — Generate HTML Report

JMeter generates a detailed HTML dashboard report from your test results.

Option A — Generate During Test Run

jmeter -n -t your_test.jmx -l results.jtl -e -o report_output

Option B — Generate from Existing Results File

jmeter -g results.jtl -o report_output

Option C — Generate from GUI

  1. Go to Tools → Generate HTML Report
  2. Select your .jtl results file
  3. Select your user.properties file
  4. Select an empty output directory
  5. Click Generate Report

Reading the HTML Report

Open report_output/index.html in your browser. Key sections to review:

SectionWhat to Look For
APDEX TableScore should be > 0.85. Lower = poor user experience.
Statistics TableCheck error %, average, 90th/95th/99th percentiles, throughput
Errors SectionIdentify error types and frequency
Response Times Over TimeShould be flat/stable. Spikes = bottlenecks.
Transactions per SecondShould increase with load. Plateau = capacity ceiling.
Active Threads Over TimeShows your ramp-up pattern
